# レイヤードアーキテクチャ規約

## 概要

アプリケーションを責務ごとのレイヤーに分離し、依存方向を一方向に制御する。

## レイヤー構成

<!-- ASSUMPTION: 4層構成を採用。プロジェクトによって層数は変わりうる -->

| レイヤー | 責務 | 依存先 |
|----------|------|--------|
| Presentation | UI・リクエスト受付・レスポンス整形 | Application |
| Application | ユースケースの実行・トランザクション制御 | Domain |
| Domain | ビジネスルール・エンティティ・値オブジェクト | なし |
| Infrastructure | DB・外部API・ファイルI/O等の技術的実装 | Domain（インターフェース実装） |

## 依存ルール

- 依存は**上位→下位**の一方向のみ許可する
- Domain レイヤーは他のレイヤーに依存しない
- Infrastructure は Domain が定義したインターフェースを実装する（依存性逆転）
- レイヤーの飛び越し（例: Presentation → Domain 直接）は禁止する

<!-- ASSUMPTION: レイヤー飛び越し禁止を厳密に適用。プロジェクトによっては許容する場合もある -->

## ディレクトリ構成例

```
src/
├── presentation/    # コントローラ、ビュー、DTO
├── application/     # ユースケース、アプリケーションサービス
├── domain/          # エンティティ、値オブジェクト、リポジトリインターフェース
└── infrastructure/  # リポジトリ実装、外部サービスクライアント
```

## 各レイヤーのルール

### Presentation

- フレームワーク固有のコードはここに閉じ込める
- ビジネスロジックを持たない
- Application レイヤーの戻り値を整形して返す

### Application

- 1ユースケース = 1クラス/関数を基本とする
- ドメインオブジェクトを組み合わせて処理を組み立てる
- トランザクション境界を管理する

### Domain

- フレームワーク・ライブラリに依存しない純粋なコードにする
- リポジトリはインターフェースのみ定義する
- ビジネスルールのバリデーションはここに書く

### Infrastructure

- Domain のインターフェースを実装する
- 外部サービスとの通信詳細を隠蔽する
- 設定値・接続情報の管理を担う
