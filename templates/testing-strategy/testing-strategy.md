# テスト方針

## テストレベル

| レベル | 対象 | 目的 |
|--------|------|------|
| ユニットテスト | 関数・メソッド単位 | ロジックの正しさを検証 |
| インテグレーションテスト | モジュール間連携 | コンポーネント間の結合を検証 |
| E2Eテスト | ユーザーフロー全体 | 実際の利用シナリオを検証 |

<!-- ASSUMPTION: テストレベルは一般的な3層構成を採用。プロジェクトによってはコントラクトテストやスナップショットテスト等の追加レベルが必要になる場合がある -->

## カバレッジ目標

| 対象 | 目標値 | 備考 |
|------|--------|------|
| ビジネスロジック | 80%以上 | 正常系・異常系・境界値を含む |
| ユーティリティ関数 | 90%以上 | 入出力が明確なため高カバレッジを維持 |
| API エンドポイント | 70%以上 | 主要パスとエラーパスをカバー |
| UI コンポーネント | 60%以上 | 主要な表示・操作を検証 |

<!-- ASSUMPTION: カバレッジ目標値は一般的なベストプラクティスに基づく推定値。プロジェクトの品質要件やリスク許容度に応じて調整が必要 -->

## テスト対象の優先度

1. **必須**: ビジネスクリティカルなロジック、決済・認証処理
2. **高**: データ変換、バリデーション、外部API連携
3. **中**: CRUD操作、UI表示ロジック
4. **低**: 設定値の読み込み、純粋な表示コンポーネント

## テストツール

<!-- ASSUMPTION: ツール選定はプロジェクトの技術スタックに依存する。以下は代表的な選択肢を列挙 -->

| 用途 | ツール例 |
|------|----------|
| テストランナー | Jest, Vitest, pytest, Go testing |
| モック・スタブ | Jest mock, unittest.mock, gomock |
| E2E | Playwright, Cypress |
| APIテスト | supertest, httptest |
| カバレッジ計測 | istanbul/nyc, coverage.py, go cover |

## テスト設計の原則

- **AAA パターン**: Arrange（準備）→ Act（実行）→ Assert（検証）
- **1テスト1検証**: テストケースごとに検証する振る舞いを1つに絞る
- **テスト名に意図を明記**: 「何を」「どうしたら」「どうなるか」を記述
- **外部依存の分離**: DB・外部APIはテストダブルで置換可能にする
- **テストデータの独立性**: テスト間でデータを共有しない

## テスト実行方針

| タイミング | 実行するテスト |
|-----------|---------------|
| ローカル開発時 | 変更に関連するユニットテスト |
| プルリクエスト | ユニット + インテグレーション |
| マージ後（CI） | 全テスト（E2E含む） |
| 定期実行 | E2E + パフォーマンステスト |

<!-- ASSUMPTION: CI/CDパイプラインの構成はプロジェクトごとに異なる。上記は一般的なフローの例 -->

## 異常系テストの観点

- 無効な入力値（null、空文字、型不一致）
- 境界値（最小値、最大値、上限+1）
- 外部サービス障害（タイムアウト、5xx応答）
- 認証・認可エラー
- 同時実行・競合状態
