# Code Review - コードレビュー

> 前提: `agents/prompts/base.md` を読み込み済みであること

## 目的

変更されたコードをレビューチェックリストに基づいて検証し、問題点・改善点を指摘する。

## 参照ドキュメント

- `docs/review-checklist/review.md` - レビューチェックリスト（主軸）
- `docs/coding-rules/nextjs/implementation-patterns.md` - 実装パターン
- `docs/coding-rules/nextjs/directory-structure.md` - ディレクトリ構成
- `docs/coding-rules/nextjs/naming-conventions.md` - 命名規則
- `docs/coding-rules/nextjs/testing.md` - テスト方針
- `docs/coding-rules/nextjs/based/coding-rules/clean-architecture.md` - 依存方向
- `docs/coding-rules/nextjs/based/coding-rules/solid-principle.md` - SOLID 原則
- `docs/security-checklist/README.md` - セキュリティチェックリスト

## 手順

### Step 1: 変更差分の取得

レビュー対象の変更差分を取得する:

```bash
# PR ベース
git diff main...HEAD

# または特定コミット
git diff <commit-hash>

# またはステージング済み
git diff --cached
```

### Step 2: 変更の概要把握

- 何の機能/修正か
- 影響範囲はどこか
- 関連するテストは含まれているか

### Step 3: チェックリストに基づくレビュー

`docs/review-checklist/review.md` のチェック項目を順に検証する:

#### 3-1. 正確性

- [ ] 要件を満たしているか
- [ ] 境界値・エッジケースが考慮されているか
- [ ] ロジックに漏れや矛盾がないか

#### 3-2. セキュリティ

- [ ] ユーザー入力のバリデーション（TypeBox 等）
- [ ] SQL インジェクション対策（パラメータ化クエリ / ORM）
- [ ] XSS 対策
- [ ] CSRF 対策
- [ ] 認証・認可のチェック（各エンドポイントで個別）
- [ ] 機密情報のハードコードがないか
- [ ] `process.env` 直接参照がないか（`lib/env.ts` 経由か）

#### 3-3. テスト

- [ ] 変更に対応するテストが追加/更新されているか
- [ ] 正常系・異常系が網羅されているか
- [ ] モックはシステム境界のみか
- [ ] テストが独立しているか（他のテストに依存していないか）

#### 3-4. エラーハンドリング

- [ ] エラーが適切にキャッチ・伝播されているか
- [ ] Domain 層は例外 throw、Server Actions は Result 型になっているか
- [ ] リソースリークがないか

#### 3-5. 可読性・保守性

- [ ] 命名規則に従っているか（`docs/coding-rules/nextjs/naming-conventions.md`）
- [ ] 単一責任の原則が守られているか
- [ ] デッドコードがないか
- [ ] 不要なコメントがないか（コードで表現できる内容をコメントにしていないか）

#### 3-6. パフォーマンス

- [ ] N+1 クエリがないか
- [ ] 不要な再レンダリングが発生しないか
- [ ] 適切なキャッシュが考慮されているか

#### 3-7. 設計・アーキテクチャ

- [ ] ディレクトリ構成に従っているか（`docs/coding-rules/nextjs/directory-structure.md`）
- [ ] 依存の方向が正しいか（外側→内側）
- [ ] feature 間の循環依存がないか
- [ ] バレルファイルを使っていないか
- [ ] Server/Client Component の使い分けが適切か

#### 3-8. Git・差分

- [ ] コミット粒度が適切か（機能変更とリファクタリングが分離されているか）
- [ ] 不要な差分（フォーマット変更のみ等）が含まれていないか

### Step 4: レビュー結果の出力

以下の形式でフィードバックを出力する:

```markdown
## レビュー結果

### 要修正（Must Fix）

| # | ファイル | 行 | 種別 | 内容 |
|---|---------|-----|------|------|
| 1 | ... | ... | セキュリティ | ... |

### 改善推奨（Should Fix）

| # | ファイル | 行 | 種別 | 内容 |
|---|---------|-----|------|------|
| 1 | ... | ... | 可読性 | ... |

### 任意（Nice to Have）

| # | ファイル | 行 | 種別 | 内容 |
|---|---------|-----|------|------|
| 1 | ... | ... | パフォーマンス | ... |

### 良い点

- ...
```

## 重要度の基準

| レベル | 基準 |
|--------|------|
| Must Fix | セキュリティ脆弱性、データ損失リスク、クラッシュ、要件未達 |
| Should Fix | ルール違反、テスト不足、可読性の著しい低下 |
| Nice to Have | より良い書き方の提案、軽微な改善 |

## 注意事項

- コードの修正はしない（指摘のみ）
- 主観的な好みではなく、ドキュメントに基づいた指摘を行う
- 良い点も積極的にフィードバックする
